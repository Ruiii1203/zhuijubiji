## 1. 产品背景

> 这里说明“为什么要做这个产品”。建议从个人真实痛点出发，讲清楚当前场景中存在什么问题，以及这个问题为什么值得被解决。

### 1.1 背景描述

作为一个日常产出剧评/影评的文字创作者，很多时候会因为记不清剧情细节，或是剧中人物的某一句台词，而无法准确地表达感悟、感受。这个时候，重新翻看剧集、定位写作素材的位置很麻烦。

所以想要设计一个产品可以随时随地记录追剧时的感悟，包括标记剧情、台词等功能。在追完整部剧后，可以自动生成一份素材库，甚至可以总结一些标记素材的共同点，反映这部作品主要在哪方面触动了我，也可以做针对某一剧中人物的前后感情对比等、剧情前后的呼应等更有趣的分析。

> 补充：这个产品本质上是在补齐“看剧时的即时灵感”与“写剧评时的素材调取”之间断裂的一环。市面上的笔记工具能记，却不懂“剧”这个上下文（剧名、集数、角色）；追剧打卡类工具懂“剧”，却不为“创作”服务。追剧笔记想要填补的，正是这两者之间的空白。

### 1.2 当前痛点

> 这里可以列出用户在现有方式下遇到的主要问题。建议控制在 3-5 条，不要写太散。

- 痛点 1：素材分散，可能以截图形式记录在手机里，也有可能只是单纯地在日记中记录了追剧过程中的零散感受
- 痛点 2：追剧后写剧评，很容易忘记追剧过程中的即时感受，重新回顾剧集很浪费时间、打断思路
- 痛点 3：写剧评时难以快速找回当时的情绪和灵感
- 痛点 4：日常追剧很难有意识地记录某一情节或台词在第几集出现，后续写剧评添加截图或片段时，寻找困难

### 1.3 产品机会

> 这里说明你认为可以通过什么产品能力改善上述问题。

通过“快速记录 + 标签管理 + 素材搜索 + 一键复制导出”，帮助用户把追剧时的灵感沉淀成可复用的创作素材。

通过“一键整理”，帮助用户把素材根据不同的形式进行集合整理，例如按照人物进行整理。

> 补充：产品的差异化机会在于把“剧集结构”作为一等公民——剧名、集数、角色不是普通标签，而是天然的组织维度。围绕“记录 → 整理 → 调取 → 创作”这条创作链路设计，而不是停留在“又一个笔记本”。

## 2. 产品目标

> 这里说明 MVP 阶段要达成什么目标。建议目标具体、可验证，避免写成过大的愿景。

### 2.1 产品定位

这是一款帮助有创作欲望的追剧人边看剧边记录感受与素材、写剧评时快速调取灵感的轻量工具。

### 2.2 MVP 目标

> 这里写第一版最重要的目标，建议 2-3 条即可。

- 目标 1：让用户能够低成本、即时地准确记录追剧感受（记录路径足够短，不打断追剧）
- 目标 2：让用户能通过剧名、角色、标签或关键词快速找到历史素材
- 目标 3：让用户能够一键导出追剧感受与关键剧情的定位信息，用于剧评创作

### 2.3 成功指标

> 这里写如何判断 MVP 是否有效。初学阶段可以先用简单指标，不需要过度追求复杂数据体系。

| 指标类型 | 指标名称 | 目标值 / 判断方式 |
|---|---|---|
| 使用效率 | 完成一条记录所需时间 | 从点击“新增记录”到保存成功 ≤ 30 秒（仅填必填项时 ≤ 15 秒） |
| 检索效率 | 找到目标素材所需时间 | 通过剧名/标签/关键词筛选，≤ 1 分钟内定位到目标素材 |
| 留存意愿 | 用户是否愿意持续记录 | 自用阶段：单部剧追剧期内连续记录 ≥ 3 天；或单部剧累计沉淀 ≥ 10 条记录 |
| 创作帮助 | 是否帮助完成一篇剧评 | 至少借助素材库 + 导出功能，辅助产出 1 篇完整剧评（公众号/小红书）；主观评价“找素材比以前快” |
| 检索成功率（补充） | 搜索后是否命中目标 | 发起搜索后，前 3 条结果中包含目标素材的比例 ≥ 80% |

> 说明：自用 MVP 阶段不必追求复杂数据体系，先用“能不能在 30 秒内记下、1 分钟内找回、最终帮我写完一篇剧评”这三个朴素问题来检验产品是否真的有用。

## 3. 目标用户与使用场景

> 这里说明产品服务谁、在什么场景下使用。建议先聚焦核心用户，不要一开始覆盖太多人群。

### 3.1 目标用户

喜欢追剧、有写剧评或人物分析习惯、会在公众号/小红书/朋友圈表达观剧感受的内容创作者。

### 3.2 用户特征

- 内容兴趣：喜欢剧情分析、人物情感体会、具有共情能力、关注情绪表达
- 创作习惯：会写剧评、人物分析等
- 记录习惯：目前使用备忘录、截图、聊天框、纸笔等
- 核心诉求：降低记录成本、提高素材复用效率、提高创作效率

### 3.3 核心使用场景

#### 场景 1：看剧中 / 看剧后即时记录

用户正在看剧，突然被某句台词、某个情节或人物状态触动，希望快速记下来。或是，在看完一部分剧集后，想快速记录阶段性的追剧感受，希望快速记录。

#### 场景 2：写剧评时调取素材

用户准备写剧评，需要按剧名、角色、标签或关键词找回过去记录的素材。

#### 场景 3：复盘角色或剧情线

用户希望查看自己对同一角色、同一剧情线在不同集数中的感受变化。MVP 可轻量支持，也可作为后续迭代。

## 4. 需求范围

> 这里明确第一版做什么、不做什么。范围控制是 PM 很重要的能力。MVP 先解决最核心的问题，不要把产品做重。

### 4.1 MVP 功能范围

| 模块 | 功能 | 简要说明 | 优先级 |
|---|---|---|---|
| 记录 | 快速记录 | 输入感受/台词/情节，手动输入剧名，可选补充集数 | P0 |
| 标签 | 打标签 | 支持默认标签与自定义标签，标签字段可留空，未填写时默认补为“感受” | P0 |
| 素材库 | 浏览素材 | 按剧名、标签等浏览记录 | P0 |
| 搜索 | 关键词搜索 | 通过关键词搜索历史记录 | P0 |
| 导出 | 批量复制/导出素材 | 同一剧目下支持批量复制/导出，导出支持纯文本与 Markdown | P0 |

### 4.2 第一版暂不做

> 这里写清楚 MVP 不包含的功能，避免需求膨胀。

- AI 自动生成对人物的感受变化/检索情节前后呼应/生成剧评功能

- 复杂富文本编辑器
- 第三方影视库接入（如豆瓣等）
- 标签颜色配置能力

### 4.3 后续迭代方向

> 这里可以放你对未来版本的想法，但不要影响 MVP 的范围。

- 方向 1：情感轨迹图
- 方向 2：素材区 + 写作区并排的创作模式
- 方向 3：拍照/视频记录
- 方向 4：标签颜色能力

### 5.1 快速记录流程

1. 用户进入产品首页
2. 点击【新增记录】
3. 手动输入剧名（必填）
4. 输入集数 / 角色 / 标签 / 台词 / 正文内容
5. 点击保存
6. 若未填写标签，系统自动补默认标签“感受”
7. 若未填写集数，系统给出轻提示“要不要补一下集数？”
8. 系统将记录沉淀到素材库

**记录界面字段说明（用户在记录时看到的字段）：**

| 字段 | 必填/可选 | 用户看到的体验 |
|---|---|---|
| 剧名 | **必填** | 手动输入框；默认带入“最近一次使用的剧名”，也可复用本地已记录过的剧名，不接入第三方影视库 |
| 标签 | **可选** | 以可点选的标签胶囊呈现，含默认标签 + “+ 自定义”入口；若未选择，保存时自动补默认标签“感受” |
| 内容 | **必填** | 多行长文本，占据主要视觉区域，进入页面即可直接输入，是记录的核心 |
| 集数 | 可选 | 数字/文本输入，可留空；保存时若为空，给出轻提示“要不要补一下集数？”，也支持后续在详情页补填 |
| 角色 | 可选 | 文本/多选，可填多个角色，可留空 |
| 创建时间 | 系统自动 | 不出现在输入区，保存时自动生成，用户无需关心 |

> 设计取舍说明：为了“不打断追剧”，**必填项收敛到 2 个（剧名、内容）**，标签改为可选并由默认标签“感受”兜底，让用户进入页面即可开打字。集数和角色虽然对后续检索有价值，但仍设为可选，避免每次记录都要做选择题；其中集数在保存时增加一次轻提示，兼顾“记录轻量”与“后续找回”的需要。

### 5.2 素材检索流程

1. 用户进入素材库
2. 通过剧名 / 标签 / 角色 / 关键词筛选
3. 浏览匹配到的素材列表
4. 点击查看详情
5. 优先执行同一剧目下的批量复制或批量导出；也支持单条操作

**“好用”的检索体验定义（写剧评时如何高效找回素材）：**

- **入口零思考**：素材库顶部常驻搜索框 + 一排剧名/标签快捷筛选，用户不必先想“我该用哪种方式找”，剧名、标签、关键词三条路都能直达。
- **组合而非互斥**：支持“剧名 + 标签 + 关键词”叠加筛选（如《XX》中“角色 A”的所有“金句”），逐步收窄，符合写剧评时“先定主题、再找具体素材”的真实思路。
- **结果可扫读**：搜索结果保留剧名、集数、标签、内容摘要，命中的关键词高亮，用户一眼能判断“是不是我要的那条”，无需逐条点开。
- **从找到到复用零摩擦**：在结果列表里就能直接批量复制/导出，不必逐条点进详情再操作，让“找素材”和“用素材”连成一步。

> 判断标准：用户写剧评时，能在不回看剧集的情况下，1 分钟内把一个主题（如某角色的成长线）相关的素材都筛出来并复制进写作工具，就算“好用”。

### 5.3 异常流程

> 这里记录用户可能遇到的异常或空状态，帮助后续设计和开发补齐体验。

| 场景 | 可能问题 | 处理方式 |
|---|---|---|
| 新增记录 | 必填字段为空 | 提示用户补充必填内容 |
| 新增记录 | 标签未填写 | 保存时自动补默认标签“感受” |
| 新增记录 | 集数未填写 | 保存时轻提示“要不要补一下集数？”，用户可继续保存 |
| 搜索素材 | 无搜索结果 | 展示空状态，并提示更换关键词 |
| 复制素材 | 复制失败 | 提示重试或手动选择文本 |
| 素材库 | 暂无记录 | 引导用户创建第一条记录 |

## 6. 功能需求说明

> 这里是 PRD 的核心部分。建议每个功能按“用户故事 - 功能说明 - 字段/规则 - 验收标准”的结构来写，既清晰又适合初学者练习。

### 6.1 快速记录

#### 用户故事

作为一个正在追剧的内容创作者，我希望能快速记录一句台词和当下感受，以便写剧评时找回灵感。

#### 功能说明

用户在产品首页，点击【新建记录】，进入填写界面，依次输入以下输入项：

1. **【剧名】** 必填，MVP 阶段先纯手动输入，不接入第三方影视库；可复用本地历史剧名，减少重复输入
2. **【集数】** 选填，可以记录是在看第几集时，新增的这份记录；若为空，保存时轻提示“要不要补一下集数？”
3. **【角色】** 选填，记录是针对哪个角色新增的记录
4. **【标签】** 选填，支持自定义；若用户未选择标签，保存时自动补上默认标签【感受】。产品自有默认标签包括：【台词】、【剧情】、【感受】、【续写】、【人物分析】
5. **【内容】** 必填，用户在这里记录情节/台词/感受/人物分析/对剧情的续写或改写等
6. **【创建时间】** 系统自动生成

#### 字段设计

| 字段 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| 剧名 | 文本 | 是 | MVP 阶段纯手动输入，支持新建剧名；不接入第三方影视库 |
| 集数 | 数字 / 文本 | 否 | 允许为空；保存时可给出补填轻提示 |
| 角色 | 文本 / 多选 | 否 | 支持多个角色 |
| 标签 | 多选 | 否 | 默认标签包括【台词】、【剧情】、【感受】、【续写】、【人物分析】，支持自定义与多选；未填写时自动补默认标签【感受】 |
| 内容 | 长文本 | 是 | 支持记录台词、感受、情节等 |
| 创建时间 | 系统生成 | 是 | 不允许用户修改 |

#### 验收标准

- 用户输入必填字段（剧名、内容）后，点击保存可以成功创建记录
- 保存成功后，记录可以在素材库中看到
- 标签为空时，系统自动补默认标签“感受”后完成保存
- 集数为空时，系统给出轻提示“要不要补一下集数？”，但不阻断保存
- 必填字段为空时，系统给出明确提示
- 用户点击素材库，发现无素材时，系统引导新建记录

### 6.2 标签管理

#### 用户故事

用户希望给素材打上“台词”“角色”“金句”等标签，以便后续按标签进行检索。

#### 功能说明

在首页的【新建记录】功能下，每一条记录都支持多选标签与自定义标签。用户也可以在素材库对标签进行更改，包括删除、新增、自定义。V0.1 支持自定义标签，但暂不开放颜色配置；若用户在创建记录时未填写标签，系统自动补默认标签“感受”。

#### 默认标签建议

- 台词
- 感悟
- 情节
- 角色
- 金句
- 伏笔
- 关系线
- 人物成长
- 人物分析
- 续写/改写

#### 验收标准

- 用户可以给一条记录添加多个标签
- 用户可以在创建或编辑记录时新增自定义标签
- 用户可以在素材库中按标签筛选记录
- 用户可以在素材库对标签进行更改
- 标签颜色在 V0.1 不开放配置，V0.2 再评估是否支持
- 若用户未主动选择标签，系统会自动补默认标签“感受”

### 6.3 素材库

#### 用户故事

用户希望集中查看所有追剧记录，以便写作前快速整理素材。

#### 功能说明

素材库默认按照剧名分组进行展示。同时，素材库的展示方式也支持自由选择，包括按照剧名分组展示、按照时间倒序显示、按照时间正序显示。暂不支持其他显示方式。同一剧目下优先支持批量复制/导出，满足写剧评前集中取材的高频场景。

#### 展示信息

| 信息 | 是否展示 | 说明 |
|---|---|---|
| 剧名 | 是 | 若有，展示在记录的左上角 |
| 集数 | 是 | 若有，展示在记录的右上角；若没有，支持点开记录后增加，增加后展示在右上角 |
| 标签 | 是 | 展示在记录的右上角 |
| 内容摘要 | 是 | 展示前 6 个字，包括字符 |
| 创建时间 | 是 | 展示在一份记录的左下角，时间格式：YYYY-MM-DD, HH:mm:ss |
| 操作入口 | 是 | 展示在一份记录的右下角，包括编辑、删除、批量复制/导出 |

#### 验收标准

- 素材库默认按剧名分组展示
- 点击素材卡片可以查看完整内容并进行编辑
- 同一剧目下可选中多条记录进行批量复制或批量导出
- 素材为空时展示引导创建的空状态

---

### 6.4 搜索与筛选

#### 用户故事

用户在写剧评时，希望通过关键词、剧名、角色或标签快速找到历史素材。

#### 功能说明

支持按照剧名/角色/标签/正文内容/时间进行搜索，支持组合筛选，搜索结果默认按照时间倒序显示，支持用户更改为按照时间正序显示。

#### 搜索范围

- 剧名：支持模糊搜索
- 角色：支持多角色筛选
- 标签：支持多标签筛选
- 正文内容：支持关键词命中
- 时间：支持按创建时间筛选

#### 验收标准

- 输入关键词后可以返回包含该关键词的记录
- 筛选某个标签后，只展示对应标签记录
- 支持在同一剧目下批量选中搜索结果并直接复制/导出
- 无结果时展示空状态提示

---

### 6.5 复制 / 导出

#### 用户故事

用户在写剧评时，希望快速复制选中素材到公众号、备忘录或其他写作工具。

#### 功能说明

支持单条复制、批量复制，也可以导出为纯文本或 Markdown。优先实现同一剧目下的批量复制/导出能力，因为该场景比单条复制更常用；单条复制作为补充能力保留。

#### 导出格式

**纯文本格式示例：**

```text
创建时间：YYYY-MM-DD, HH:mm:ss
《剧名》第 X 集
标签：台词 / 角色 / 感受
内容：这里是记录的原文
```

**Markdown 格式示例：**

```markdown
### 《剧名》第 X 集
- 创建时间：YYYY-MM-DD, HH:mm:ss
- 标签：台词 / 角色 / 感受
- 内容：这里是记录的原文
```

#### 验收标准

- 用户可以对同一剧目下的多条素材进行批量复制
- 用户可以对同一剧目下的多条素材进行批量导出
- 用户点击复制后，素材内容成功进入剪贴板
- 复制成功后出现轻提示
- 导出格式同时支持纯文本与 Markdown
- 复制/导出内容格式清晰，适合直接粘贴到写作工具

## 8. 交互与体验原则

> 这里写产品体验上的设计原则，帮助后续做原型和开发时保持一致。

### 8.1 轻量记录

支持快速保存，默认带入最近剧名；标签可不填，由系统自动补默认标签“感受”。

### 8.2 不打断追剧体验

产品应避免复杂操作，记录路径尽量短，适合边看剧边使用；集数只做轻提示，不做强阻断。

### 8.3 写作时快速调取

素材库和搜索体验要围绕“写剧评时找素材”设计，而不是单纯做笔记归档；同一剧目下的批量复制/导出优先于单条操作。

### 8.4 情绪与创作氛围

结合产品特色，主视觉需要简洁且有电影感、光影感、情绪感、创作感，文案以简洁为主。

## 9. 数据与内容结构

> 这里定义一条“追剧笔记”在系统里包含哪些数据。即使是轻量产品，也建议提前想清楚数据结构，方便后续开发。

### 9.1 笔记数据结构

| 字段名 | 含义 | 示例 | 备注 |
|---|---|---|---|
| id | 笔记唯一标识 | note_001 | 系统生成 |
| drama_name | 剧名 | "去有风的地方" | 必填 |
| episode | 集数 | 12 | 可选 |
| character | 角色 | ["许红豆", "谢之遥"] | 可选，数组形式支持多个角色 |
| tags | 标签 | ["感受"] | 可选，数组形式支持多个标签；若为空则保存时自动补默认标签 |
| content | 记录内容 | "她说'有些路，只能一个人走'，这句台词戳到我了" | 必填 |
| created_at | 创建时间 | "2026-06-16 21:35:08" | 系统生成 |
| updated_at | 更新时间 | "2026-06-16 21:35:08" | 系统生成 |

> 补充：`character` 与 `tags` 建议用数组（多值）存储，方便后续做“按角色复盘”“按标签筛选”；`episode` 用字符串/数字均可，留空表示未填；`tags` 在入库时应保证至少有一个值，若用户未填写则由系统补默认标签“感受”。一条完整记录的 JSON 示例：

```json
{
  "id": "note_001",
  "drama_name": "去有风的地方",
  "episode": 12,
  "character": ["许红豆"],
  "tags": ["感受"],
  "content": "她说'有些路，只能一个人走'，这句台词戳到我了",
  "created_at": "2026-06-16 21:35:08",
  "updated_at": "2026-06-16 21:35:08"
}
```

### 9.2 标签数据结构

标签支持自定义，V0.1 暂不支持颜色配置，V0.2 再评估颜色能力。

> 补充：MVP 阶段标签可以先作为字符串数组直接存在笔记里，不必单独建表；V0.1 先满足“默认标签 + 自定义标签 + 自动兜底默认标签”，待“颜色 + 更复杂标签管理”需求变强时，再抽出独立的标签表（`tag_id / name / color / is_default`）。

### 9.3 本地存储 / 后端存储设想

**MVP 推荐方案：浏览器本地存储（localStorage / IndexedDB），无后端。**

适合 Vibe Coding MVP 的渐进式选型建议：

| 阶段 | 方案 | 适用情况 | 优缺点 |
|---|---|---|---|
| **第一步（强烈推荐）** | 浏览器 `localStorage` | 自用、数据量小（几百条以内）、纯前端单页应用 | 零后端、零成本、改起来最快，适合 Vibe Coding 快速验证；缺点是仅限本机本浏览器，换设备/清缓存会丢 |
| 进阶 | 浏览器 `IndexedDB`（可配 Dexie.js） | 记录变多、需要更结构化的查询/搜索 | 仍是纯前端，支持更大数据量和更快检索；缺点是 API 略复杂 |
| 需要多端 | 轻量云端 BaaS（如 Supabase / LeanCloud / 飞书多维表格 API） | 想换设备也能用、想加简单备份 | 多端同步、有基础后端能力；缺点是引入账号体系与网络依赖 |
| 数据备份兜底 | 本地 JSON 导入/导出 | 任何阶段 | 提供“导出全部笔记为 JSON / 导入 JSON”能力，作为换设备和防丢失的低成本兜底 |

> 结论：**V0.1 先用 `localStorage`（搭配 JSON 导出做兜底）**，把精力放在“记录 → 检索 → 复用”链路本身；等真正出现“想换设备继续用”的需求时，再升级到 IndexedDB 或云端方案。这样既符合 MVP“最小可用”的原则，也避免一上来就背上后端的复杂度。

## 10. 非功能需求

> 这里写不是具体功能、但会影响体验的要求。轻量 PRD 保留最关键几项即可。

### 10.1 性能要求

- 首页 / 素材库页面首屏加载时间 ≤ 2 秒（本地存储方案下应基本秒开）
- 搜索/筛选响应时间 ≤ 500 毫秒（数百条记录规模内做到“输入即出结果”）
- 新增记录保存后 ≤ 1 秒内在素材库可见，操作有即时反馈，不卡顿
- 单条记录复制操作 ≤ 1 秒完成并给出轻提示
- 同一剧目下批量复制/导出操作响应时间可接受，优先保证常见批量场景顺畅完成

### 10.2 兼容性要求

- **优先支持移动端浏览器**（追剧场景多发生在手机/平板上，边看边记），主流机型上的 Safari / Chrome 正常使用
- 兼顾桌面端浏览器（Chrome / Edge / Safari 最新版本），用于写剧评时在电脑上整理与复制素材
- 采用响应式布局，保证手机竖屏与桌面宽屏下都能正常展示
- MVP 阶段不强制支持 IE 等老旧浏览器

### 10.3 可用性要求

- 按钮文案清晰、空状态有引导、保存后有反馈
- 标签缺省时自动兜底，避免用户因未选标签而被阻断
- 集数缺省时只做轻提示，不影响保存完成

### 10.4 隐私与数据安全

- 用户记录内容默认仅自己可见

## 12. 验收标准汇总

> 这里把 MVP 是否完成的判断标准集中列出，方便开发完成后自测。

### 12.1 核心链路验收

- 用户可以成功创建一条追剧笔记
- 用户可以在素材库看到已创建的笔记
- 用户可以通过关键词搜索到目标笔记
- 用户可以按标签或剧名筛选素材
- 用户可以对同一剧目下的多条素材执行批量复制或批量导出
- 用户可以导出 Markdown 格式素材用于写作

### 12.2 异常状态验收

- 必填字段为空时有提示
- 标签为空时自动补默认标签“感受”
- 集数为空时出现轻提示，但不阻断保存
- 搜索无结果时有空状态
- 素材为空时有新建引导
- 复制成功或失败都有反馈

### 12.3 体验验收

- 记录流程足够短，不打断追剧
- 素材列表可读性强，能快速识别内容
- 复制/导出后的内容格式适合直接进入写作工具
- 用户在批量整理同一剧目素材时，操作路径短、理解成本低

## 13. 风险与待确认问题

> 这里记录当前还没想清楚的问题，帮助你后续逐步完善 PRD。PM 文档不需要一开始完美，但需要把不确定性显性化。

### 13.1 产品风险

- 风险 1：标签体系过多导致选择负担
- 风险 2（补充）：纯本地存储下，换设备或清缓存会丢失数据，需要靠导出兜底提醒用户
- 风险 3（补充）：“边看剧边记录”要求记录路径极短，若表单字段或交互过重，用户会退回用备忘录/截图，产品价值落空

### 13.2 待确认问题

- 问题 1【已确认】：标签字段改为可选；当前必填项调整为“剧名 + 内容”，若用户未填写标签，系统自动补默认标签“感受”。
- 问题 2【已确认】：剧名在 MVP 阶段先纯手动输入，不接入第三方影视库；可复用本地历史剧名，但不依赖外部影视数据源。
- 问题 3【已确认】：集数字段继续保留可选；用户保存时若未填写，系统给出轻提示“要不要补一下集数？”，但不阻断保存。
- 问题 4【已确认】：V0.1 支持自定义标签，但暂不开放颜色功能；颜色能力放到 V0.2 再评估。
- 问题 5【已确认】：导出格式除纯文本外，还需支持 Markdown，方便贴入公众号或其他写作工具。
- 问题 6【已确认】：批量复制/导出的优先级高于单条复制；同一剧目下的批量操作是更常用场景，优先实现。
